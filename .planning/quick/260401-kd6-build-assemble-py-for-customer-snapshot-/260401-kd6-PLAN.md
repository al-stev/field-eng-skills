    Create `assemble.py` in `.claude/skills/customer-snapshot/templates/` as a CLI script:

    **CLI interface:**
    ```
    python assemble.py --customer "GResearch" \
      [--jira /path/to/jira.json] \
      [--bq /path/to/bq.json] \
      [--asana /path/to/asana.json] \
      [--sentiment /path/to/sentiment.json] \
      [--days 14] [--months 6] [--audience internal] \
      [--output /path/to/output.json]
    ```
    All data source args are optional. Output goes to stdout by default, or to --output file path.

    **Core function:** `assemble_intelligence_data(customer, jira_data, bq_data, asana_data, sentiment_data, config)` returns complete INTELLIGENCE_DATA dict.

    **Normalization maps:** Copy COMPONENT_NORMALIZE and PARENT_NORMALIZE exactly from issues.js (lines 56-99). These are Python dicts with identical keys and values. The JS maps are the source of truth.

    **Theme assignment** (per issue, 3-tier cascade):
    1. If issue has components, take first component. Look up in COMPONENT_NORMALIZE. If found, use normalized value as theme.
    2. Else if issue has parent_summary, look up in PARENT_NORMALIZE. If found, use normalized value as theme.
    3. Else assign "Uncategorized".

    **Priority mapping:** Map Jira priority names to P-codes: Critical->P0, Highest->P0, High->P1, Medium->P2, Low->P3, Lowest->P3. If already P0-P3, passthrough. If None, keep None.

    **Jira issue transformation:** Transform each raw Jira issue from issues.py format to INTELLIGENCE_DATA.issues format:
    - Copy: key, summary, status, assignee, created, updated, resolutiondate, url, components, parent, parent_summary, comments
    - Map: type (keep as-is), priority (apply P-code mapping), theme (apply cascade)

    **Asana task transformation:** Transform raw Asana tasks to INTELLIGENCE_DATA.actions format:
    - Compute overdue: due_on < today and not completed
    - Compute stale: (today - modified_at) > 7 days AND section in ["To Do", "In Progress"]
    - Compute stale_days: days since modified_at
    - Extract linked_jira from task name via regex `\(WB-\d+\)` -> "WB-XXXX"
    - Extract priority from custom_fields Priority enum: High->P1, Medium->P2, Low->P3
    - Extract section from memberships[0].section.name
    - Build summary counts (total, in_progress, waiting, todo, overdue, stale)

    **Trending computation** (from transformed issues):
    - opened_by_month: count issues by created month for last N months (config.trending_months, default 6)
    - closed_by_month: count resolved issues by resolutiondate month
    - raised_to_resolved_ratio: total opened / total closed in period (null if zero closed)
    - median_ttr_days: median (resolutiondate - created) for resolved issues in period
    - theme_recurrence: top 5 themes by issue count with counts

    **Graceful degradation:** Each missing input produces an appropriate stub:
    - No jira: issues=[], trending=null
    - No bq: usage={"available": false, "reason": "not_provided"}
    - No asana: actions={"available": false, "reason": "not_provided"}
    - No sentiment: sentiment=null

    **Output shape:** Must match INTELLIGENCE_DATA schema exactly as defined in SKILL.md Step 7. Include: customer, generated (today's date), config, issues, sentiment, trending, exec_summary (null -- computed client-side), actions, usage.

    No new dependencies needed -- only stdlib (json, argparse, datetime, statistics, re, pathlib). pyyaml already in pyproject.toml but not needed here (customers.yaml is read by the caller, not assemble.py).
